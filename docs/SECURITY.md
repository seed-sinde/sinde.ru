# Безопасность

## Куда сообщать об уязвимостях

Публично уязвимости не публикуются.

Сообщать нужно письмом на [bug@sinde.ru](mailto:bug@sinde.ru).

В сообщении желательно указать:

- краткое описание проблемы;
- шаги воспроизведения;
- ожидаемое и фактическое поведение;
- оценку воздействия, если она понятна;
- логи, скриншоты или proof of concept, если они есть.

## Что считается чувствительными данными

В публичный репозиторий нельзя добавлять:

- реальные `.env` файлы;
- production credentials;
- приватные ключи;
- токены доступа;
- боевые пароли к БД, Redis, MinIO, SMTP и другим сервисам.

В репозитории допустимы только шаблоны вроде `.env.example` без реальных значений.

## Область действия

Политика распространяется на:

- backend API;
- frontend приложение;
- аутентификацию, сессии и 2FA;
- обработку пользовательских файлов;
- публичную документацию и конфигурационные шаблоны.

## Базовые правила для contributors

- не публиковать уязвимости в issues и discussions;
- не добавлять секреты даже временно;
- перед push проверять diff на случайные `.env`, ключи и дампы;
- если секрет уже попал в git-историю, его нужно ротировать, а не только удалять из файлов.
